Configuration

hm depends on a configuration file to specify the Devices with which to communicate. It also specifies Resources that enable you to use names that refer to options that are used by hm. The default configuration file name is ‘hm.conf’. The config file name can be changed by using the -c switch when Starting hm.

Comments can be entered in the configuration file using the ‘#’ character. Anything on the same line after the ‘#’ character is a comment. Blank lines are also ignored.

Devices

hm needs to know what devices to communicate with. Currently, hm communicates with devices using either a serial interface or a file interface (such as Aerodyne analyzers). Devices are specified in the configuration file using the format

device devicename interface device_file 0 baud

where

  • ‘device’ is the literal word ‘device’

  • ‘devicename’ is a name assigned to the device

  • ‘interface’ is the word ‘serial’

  • ‘device_file’ is the linux device file for the serial interface

  • ‘baud’ is the baud rate for the serial interface

For example, we want to communicate with an Agilent 34970 Data Acquistion/Control device, which is connected to a serial interface using the device file ‘/dev/ttyS0’ at 57600 baud. You would enter a line in the configuration file with:

device HP34970 serial /dev/ttyS0 0 57600

Each device must have a unique name, so if you have multiple devices of the same type, you can append an ‘.n’ to the name to distinguish them, e.g. if we added a second 34970 device, add a line:

device HP34970.2 serial /dev/ttyS1 0 57600

The device name will be the name used for the device in all ‘action lines’ used by hm.

For devices where the interface is file, the configuration line would look like

device aerodyne file /path/to/file 0 0

Virtual Devices

Certain devices can be daisy-chained together on one interface. For example, multiple valco stream selection valves can be connected together and have only one serial interace to the computer. Each valve has a unique id setting to distinguish it from the others. In cases like this, we need to define each valve on the chain as a ‘virtual device’. This can be done by setting a line in the configuration file for each valve, with a unique name and options for the valve. The configuration line looks like

virtualdevicename devicename options

For example suppose we have a device definition line in our configuration file:

device ValcoValves /dev/ttyUSB1 0 9600

Then we define each valve as a virtual device with:

StdSelectValve ValcoValves multiposition 0 8
SampleSelectValve ValcoValves multiposition 1 4
where the fields are

  • virtual device name

  • device name

  • type of valve

  • valve id number

  • number of positions

When creating action lines, the virtual device name can be used instead of the actual device name, e.g.

0 TurnValve StdSelectValve 3

The StdSelectValve field will be replaced with the actual device name, and the valve id number will be used when communicating with the valve.

Modules

Because the software is broken into multiple subclasses depending on instrument, hm needs to know which module to use for communicating with the instrument. Modules are specified in the configuration file using the format

module devicename modulename

where

  • ‘module’ is the literal word ‘module’

  • ‘devicename’ is a name assigned to the device

  • ‘modulename’ is a python module name assigned to the device (without the .py extension). See ‘modules’ section.

For example, the HP34970 device shown in the example above needs a line in the configuration file

module HP34970 scpi

because the HP34970 device uses the SCPI protocal for communications.

Note

If a module is not defined for a device, the generic ‘hmsubs’ module is used.

Resources

Another thing the configuration file can do is to assign a name to various option values. This is called a ‘resource’. This name is then used in the ‘action lines’ by hm. This way, you can modify the option values in just one place, the configuration file.

A resource is basically a ‘name value’ pair. For example, let’s say we want to assign the name ‘W1’ to relay number 101 on a device. We would enter:

W1 101

in the configuration file. Now anytime we create an action line for this relay, we use the keyword @W1 instead of the number 101, i.e.

0 CloseRelay HP34970 @W1

The value part of the resource does not need to be a single word. Some actions require multiple options expressed as a space delimited string. For example, the ConfigChannel action takes an option that requires both a location and type values. This can be specified as a resource with

Licor 101 DCVOLT

and used in an action line with

0 ConfigChannel HP34970 @Licor

which would configure channel 101 on the HP34970 to read DC volts.
